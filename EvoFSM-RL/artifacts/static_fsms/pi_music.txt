CATEGORY: ADD_ENTRY
  precondition: App open on the list screen where new items of this type are created
  abstract_steps:
    1. Locate the primary add/create control (often a floating action button or + icon)
    2. Trigger it to open a creation dialog/form
    3. Fill required fields (e.g., name)
    4. Confirm via the positive dialog button
  failure_modes:
    - 'Clicking the add control on the wrong tab/section'
    - 'Confirming with empty or invalid name'
    - 'Dismissing the dialog instead of confirming'
  verification_checklist:
    - 'New item is visible in the corresponding list after dialog closes'
    - 'Item name matches input exactly'
    - 'No error toast/dialog remained'

CATEGORY: NAVIGATE_TO_SECTION
  precondition: App open with tabbed or drawer navigation available
  abstract_steps:
    1. Identify the section name relevant to the task (distinguishing between the different item-type sections available in the app)
    2. Click the corresponding tab or drawer entry
    3. Wait for the list content to load
  failure_modes:
    - "Acting on the wrong tab's content"
    - "Assuming an item exists in a section it doesn't belong to"
  verification_checklist:
    - 'Tab indicator reflects chosen section'
    - "Header/count matches expected section (e.g., 'N <ItemType>')"

CATEGORY: QUERY_INFO
  precondition: Relevant list or detail view is reachable and not empty
  abstract_steps:
    1. Navigate to the section containing the answer
    2. Optionally apply a sort/filter to expose the required ordering
    3. Read the requested field(s) directly from the UI
    4. Return an answer action with the extracted value
  failure_modes:
    - "Fabricating data when the target isn't in the library"
    - 'Reading without ensuring correct sort direction'
    - 'Confusing counts across different sections (mixing up item-type totals)'
  verification_checklist:
    - 'Value is visible on-screen at time of answer'
    - "Sort/filter state matches the question's requirements"
    - 'If item cannot be found after a bounded search, declare infeasible'

CATEGORY: SEARCH_FOR_ITEM
  precondition: A search affordance is visible on a list screen
  abstract_steps:
    1. Open the search interface
    2. Type a concise query (a single attribute, not multiple attributes concatenated)
    3. Inspect results; if none, try a broader/alternate term once
    4. If still empty, conclude the item is absent
  failure_modes:
    - "Appending new text to existing query (producing garbage like 'QueryQuery')"
    - 'Retrying the same query repeatedly'
    - 'Treating empty results as transient and looping'
  verification_checklist:
    - 'Query field shows the intended text only'
    - "Either a result row appears or an explicit 'No results' message is shown"
    - 'Bounded retry count respected before declaring infeasible'

CATEGORY: SORT_LIST
  precondition: On a list screen that exposes a sort control
  abstract_steps:
    1. Open the ordering configuration dialog via its icon
    2. Select the desired attribute
    3. Explicitly toggle the ordering direction to the required state
    4. Dismiss dialog and verify order
  failure_modes:
    - 'Selecting only the attribute without setting direction'
    - "Using the wrong list's sort dialog (attributes differ across sections)"
    - 'Declaring success without visually confirming the new order'
  verification_checklist:
    - 'Dialog shows target attribute marked'
    - 'Direction toggle reflects target direction'
    - 'Top-most items in list match expected ordering'

CATEGORY: OPEN_CONTAINER
  precondition: A container item is listed
  abstract_steps:
    1. Tap the container row
    2. Verify detail view loads
    3. Check whether the container is empty before further actions
  failure_modes:
    - 'Attempting interaction on an empty container'
    - 'Treating an empty-container message as a transient error'
  verification_checklist:
    - 'Detail view title matches selected container'
    - 'If empty, an explicit empty-state message is present; declare infeasible for content-dependent goals'

CATEGORY: PLAY_MEDIA
  precondition: Target media item is visible in a list or detail view
  abstract_steps:
    1. Locate the target row
    2. Tap to start playback or use the row's play control
    3. Confirm playback UI (now-playing bar / player screen) appears
  failure_modes:
    - 'Target not in local library (infeasible)'
    - 'Tapping a container instead of a playable item'
  verification_checklist:
    - 'Now-playing indicator reflects correct item identity'
    - 'Playback controls (pause/seek) are operable'
