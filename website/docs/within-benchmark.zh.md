# 基准内（AndroidWorld+）

基准内研究在单一基准上考察核心的测试时自适应（TTA）问题：在一组源 app 上训练后，当 agent 在小预算上自适应时，它在**同一基准的留出 app** 上能提升多少？这是隔离 EvoFSM 各组件贡献的最干净设定，因为训练和测试 app 来自同一套构建流程，共享相同的任务 schema、动作空间和评分 harness。

## 设置

**基准。** AndroidWorld+ —— 在 Google 的 AndroidWorld 基础上扩展了来自 BMOCA 和 AndroidLab 的 6 个 app，共 **25 个 app**。基座 agent 是 Qwen3-VL-8B-Instruct，由 M3A 两阶段循环驱动（逐步选动作 → 执行 → 逐步总结）。

**三个互斥划分。** app 和任务模板被切分开，使预训练或自适应阶段见过的任何东西都不会在评估时重现：

| 划分 | 作用 | 包含内容 |
|---|---|---|
| **Pool** | Phase-1 预训练 | 源 app；其成功轨迹训练共享 LoRA，并播种每类别库 `L_C`。 |
| **Category** | 类别结构 | 用于构建和路由 `L_C` 的 Play-Store 类别分组（每类别一个库）。 |
| **Template** | 自适应 vs. 评估 | 对每个目标 app，任务模板切分为 `T_adapt`（用于自适应）和**与之模板互斥**的 `T_eval`（仅用于评分）。 |

目标 app 相对源 pool 分为两个迁移层级：

- **Tier-B（近迁移）** —— 目标 app 的类别*确实*在源 pool 中有代表，因此存在匹配的 `L_C`。
- **Tier-C（远迁移）** —— 目标 app 的类别*不在* pool 中。没有匹配的 `L_C`，因此在静态注入下提示词与零样本基线逐字节相同。**Tier-C 因此是注入机制的内建空对照（null control）。**

**评估协议（`T_eval`，K=3）。** 35 个冻结的 `T_eval` 模板（Tier-B 18 个、Tier-C 17 个）× **K=3 个种子** = 每条线 **105 个 episode**。成功判定为 AndroidWorld 在终止时基于规则的 `is_successful(env)`；自报的 `status:complete` 本身不算数。

## 消融阶梯

四级，每级在前一级之上增加一项能力：

- **B1** —— 零样本基座模型。不注入 FSM 知识，不更新权重。
- **B2** —— `+ 静态 L_C`。把冻结的每类别抽象动作库（由 Claude Opus 从源 pool 聚合而成）注入到选动作提示词中。FSM 不演化，LoRA 不变动。
- **B3** —— `+ FSM 演化`。从 B2 的静态 `L_C` 出发，在目标 app 的 `T_adapt` 上对库做在线变异，冻结后再评估。权重仍不变动。
- **B4** —— `+ 联合 LoRA`。完整的 EvoFSM 方法：演化后的 FSM **和**每 app 的 LoRA（由 Phase-1 的 π^pre 初始化）在 `T_adapt` 上协同自适应，随后冻结用于 `T_eval`。

![AndroidWorld+ 消融阶梯 —— B1 到 B4 整体 T_eval](assets/rl_ladder.png){ width="620" }

| 级别 | 方法 | Tier-B | Tier-C | **整体（T_eval）** |
|---|---|:---:|:---:|:---:|
| **B1** | 零样本 Qwen3-VL-M3A | 47.2 | 29.4 | **38.6** |
| **B2** | + 静态类别 `L_C` | 56.5 | 29.4 | **43.3** |
| **B3** | + 每 app FSM 演化 | 63.9 | 31.4 | **48.1** |
| **B4** | + 联合 LoRA（per-tier π^pre） | 70.4 | 34.3 | **52.9** |

*（所有数字为 105 个 episode 上的平均成功率；每条线 Tier-B n=54，Tier-C n=51。）*

### 撑起结论的增量

把阶梯看作叠在零样本地板上的两个自适应通道：

- **符号通道 —— B1 → B3 = 整体 +9.5 pp**（38.6 → 48.1）。几乎全部落在**未见过 app** 的层级（Tier-B 47.2 → 63.9）；远迁移几乎不动。这一分化是因果而非偶然：在 **Tier-C 上静态步 B2 − B1 = +0.0 pp** —— 空对照精确地按设计触发，没有匹配的 `L_C`，B2 提示词与 B1 逐字节相同，两条线都收敛到 29.4%。所以增益追踪的是 `L_C` 的*内容*，而非运行条件。
- **权重通道 —— B3 → B4 = 整体 +4.8 pp**（48.1 → 52.9）。在演化后的上下文之上，per-tier π^pre 的 LoRA 在 `T_adapt` 上协同自适应，带来进一步增量，两个层级齐升（Tier-B 63.9 → 70.4，Tier-C 31.4 → 34.3）。

联合方法的两个通道在 AndroidWorld+ 上都有贡献 —— 符号通道占较大份额，权重通道在其之上再添一份增量。

### 在哪里有帮助

总体增益集中在那些源 pool 类别与目标有强结构重叠的 app 上：

- **`pro_expense` 33% → 67%**（B2 下）—— 由 Finance `L_C`（从 `bluecoins` 合成）驱动。静态注入扫描中单 app 提升最大。
- **`system_settings` 75% → 92%**（到 B3）—— 先是 Tools `L_C`（来自 `calculator` / `clock` / `files`），再经 Wi-Fi/Bluetooth 开关 `T_adapt` 模板上的演化，朝着 `…Verify` 模板所需的精确验证模式锐化。这 `+11.1 pp` 是最强的单个 B3 信号（n 最大的 Tier-B app，n=18）。

另一面是结构性的：从列表视图 app 合成的静态库，在概览屏*看起来*像列表但截断了内容的 app 上可能误触发（`simple_calendar_pro` 月视图案例），而有些失败（`chrome/Browser*` 的 `open_app` 幻觉、`camera` 视频拍摄手势时序）完全位于 `L_C` 上游，符号自适应和权重自适应都无法修复。

!!! note "B4 = 52.9% 是什么"
    B4 是唯一两个通道都在线的一级。这个醒目数字综合了三个自适应面 —— per-tier π^pre 的 LoRA 初始化、GRPO 权重更新、以及每 app 演化的 `L_C` —— 在 `T_adapt` 上协同自适应，随后冻结用于留出的 `T_eval`。它位于阶梯顶端 **52.9%**，比仅符号的 B3 高 +4.8 pp。

## 要点

1. **符号先验在干活。** B1 → B2 → B3 是干净、单调的上升（38.6 → 43.3 → 48.1），Tier-C 持平的空对照让我们有信心相信 Tier-B 增益是因果的而非噪声。
2. **静态知识有天花板；演化抬升它。** B2 是冻结的类别级库所能买到的；B3 的 `+3.7 pp`（集中在 `system_settings`）是在线变异在其之上补回的。
3. **权重通道再添一份增量。** 在演化后的上下文之上，联合 LoRA（per-tier π^pre）把 **B3 → B4 抬升 +4.8 pp**（48.1 → 52.9），两个层级齐升（Tier-B 63.9 → 70.4，Tier-C 31.4 → 34.3）。同一权重通道在更难的跨基准场景下是否同样显现，见[跨基准](cross-benchmark.md)研究。
4. **看逐 app 拆解，而不只是总体。** 在 K=3 下每条线的 episode 数较小，因此大而一致的逐 app 跃升（`system_settings`、`pro_expense`）与总体均值一道，才是诚实的报告单元。
